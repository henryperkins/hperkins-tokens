WordPress 6 · shipped plugin

AI Provider for Codex

A local-runtime WordPress AI Client provider where Codex and ChatGPT authentication stay outside WordPress while the provider still behaves like a normal connector.

Artifacts

case study

release

v2.0

ideation

Read artifact

Problem

When a WordPress site uses an external AI runtime, the architectural question is not the model. It is where authentication, billing, runtime state, and provider integration live. A site-wide API key in the database would turn one user's account into shared infrastructure and make WordPress the token store.

Build

The plugin keeps WordPress on the provider side: connector registration, settings, cached account snapshots, and per-user metadata. A localhost sidecar owns Codex runtime behavior, device-code login, and per-user account state. WordPress talks to loopback HTTP with bearer auth; the browser never talks to the runtime directly.

The architecture came from an AI-assisted divergence pass: direct calls, WordPress-stored tokens, hosted broker, callback OAuth, and local sidecar. The sidecar survived because it satisfied the hard constraints with the smallest boundary, then release verification made that boundary executable.

Outcome

  1. Boundary enforced: six tagged releases through v2.0, with release checks and Plugin Check coverage.
  2. Isolation by construction: no Codex or ChatGPT credentials are stored in WordPress; each user connects their own account.
  3. Capability growth without boundary drift: v2.0 adds capability-gated Codex image generation while keeping auth and generated artifacts on the runtime side.
  4. Scope stated honestly: self-managed hosts, local runtime, text plus gated image support.

WordPress 6 · live client work

Voices of Judah

A booking-first site for a working gospel ministry, rebuilt as one Cloudflare Worker and one booking endpoint after the first design taught what the audience actually needed.

A gospel ministry's front door is a booking request. DJ Lee and Voices of Judah needed churches, families, and event planners to get from "found you" to "request sent" on one screen, with the request landing reliably in the ministry inbox.

The second design is a single Cloudflare Worker serving a static front, with one server-side endpoint, /api/booking, that posts directly to the ministry inbox path. No form vendor sits between the visitor and the ministry, so the privacy promise is backed by the code path.

  1. One stack choice, two shipped designs: the iteration trail is public and diffable.
  2. Privacy by construction: the only dynamic surface is the booking endpoint.
  3. Traceable booking path: Worker observability keeps each submission measurable.
  4. Light front door: the interactive page stays small because the site is booking-first, not CMS-first.

More than a decade of WordPress work is what qualifies the stack decision: the right move for this client was not WordPress. Right-sizing the platform is fluency, not disloyalty.

WordPress's AI features look scattered until you see the trust model underneath. The Abilities API exposes WordPress as discoverable, schema-described capabilities an agent can invoke. A governance plane, including Connector Approvals and request logs, governs what those capabilities may do and records calls inside the owner's system. A provenance layer can attest to output after it leaves the site.

The AI Leaders connection is measuring impact. Real impact is what survives an instrument the author does not control. In this stack, the instruments are schema, log, and signature: inspectable capabilities, auditable use, and origin claims that can travel outside the site.

The work pattern is the same one this site uses: do not assert more than the evidence can carry, read status from public instruments, and correct the page when the instrument changes.

Pippin Williamson built Sandhills Development around GPL WordPress plugins such as Easy Digital Downloads, AffiliateWP, and WP Simple Pay, then sold the portfolio to Awesome Motive in 2021. The lesson is not that free code removes revenue. It relocates revenue to support, extensions, service layers, trust, and assets that cannot be copied from a plugin zip.

The Payouts Service was the key signal in the research: a hosted, usage-based layer around an open plugin ecosystem. That maps directly to the Flavor Agent thesis. The free plugin can be the adoption path, but durable value concentrates in the governed server-side layer: audit, safe reversal, external-agent controls, and evidence the reader can inspect.

The release-readiness artifact separates two audiences. The product-coherence question asks whether Flavor Agent knows what it is. The WordPress.org reviewer question asks whether it complies with plugin-directory rules. Those are different gates, and the artifact treats them as separate tracks.

  • Submission artifacts: reviewer-facing readme, disclosure inventory, reproducible zip, and current changelog.

AI Leadership 6 · agency and adaptability

Developer Docs AI Search

A system for staying current that turns Cloudflare AI Search into shared WordPress AI infrastructure for both Flavor Agent and my own agent-assisted development workflow.

The monitoring system is source-driven: official docs, GitHub release notes, WordPress AI updates, Cloudflare documentation, OpenAI/Codex changes, Claude Code/Codex workflows, and implementation-heavy community conversations.

The filter is whether a tool removes a constraint in work already shipping. Cloudflare AI Search passed because it helps Flavor Agent ground recommendations in current WordPress developer documentation and gives the agent workflow I use while coding the same fresh corpus.

  1. Discover: monitor trusted sources close to the work.
  2. Spike: test with dry runs, trusted URLs, validation queries, and controlled update paths.
  3. Measure: source freshness, coverage, retrieval quality, exact-symbol behavior, cost, latency, privacy, and debugging clarity.
  4. Decide: integrate, keep optional, or reject; every integration must stay reversible.

The lesson is that staying current is an integration discipline, not a reading habit. A new AI tool earns its place only when it can be wrapped in validation, rollback, and source provenance.

Flavor Agent · governed demo dossier

Governed AI Demo

The live MCP and wp-admin evidence package behind the Flavor Agent governance story: AI proposes, WordPress approves, every change is bounded, attributed, and reversible.

Artifacts

case study

Flavor Agent

Read artifact

The demo frame is the enterprise problem, not the plugin: an AI agent pointed at a live publishing system is unbounded, unattributable, and hard to reverse. Flavor Agent turns those into properties a reviewer can see.

  1. Bounded operations: the universal MCP surface exposes read and preview tools; write-capable tools live behind a dedicated server.
  2. Review gate: external style apply creates a pending activity row; there is no agent approval tool.
  3. Server-side attribution: Settings > AI Activity records who requested what, through which ability, provider, and route.
  4. Fresh approval: human approval rechecks the baseline before applying.
  5. Drift-safe undo: reversal restores only when live state still matches the recorded post-apply state.

The portable thesis is simple: any system that lets AI act on live state needs bounded operations, gated review, attribution, and provable rollback. I built it on WordPress because that is my depth; the pattern travels.
