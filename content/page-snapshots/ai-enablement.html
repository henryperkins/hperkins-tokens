Expose, Govern, Attest

Reading WordPress’s AI stack as a trust model

WordPress’s AI features look scattered until you see the trust model underneath: capabilities an agent can inspect, use that the owner can govern, and provenance an outsider can verify.

Henry Perkins

Updated 22 June 2026

I came up through WordPress’s earlier democratizations, and they rhyme. In 2012, page builders dropped the floor — you assembled a site instead of hand-coding one, and I was running the community around one of them. Blocks made the same move around 2018: assemble from parts, not markup. AI is that instinct one level up — describe the work and point an agent at it. Every wave widened who could build and, in the same stroke, produced more fragile sites, because dropping the floor removes the friction that used to force inspection. I’ve watched it happen twice, so I read this wave for the thing that decides whether it holds: the discipline someone deliberately puts back after the friction is gone.

WordPress’s AI stack stops looking scattered once you read it as three moves, each answering to a different person. Expose the capability, and an agent can find it. Govern the use, and the owner stays accountable for it. Attest the output, and a stranger can check it after it leaves the site. Read together, they’re one trust model in three widening rings — capability, site, artifact. That synthesis is mine: a direction I read across the public roadmap, not a name the project has adopted.

This document is a portfolio artifact: a stranger can verify its claims even off-site.

Ring I · Capability

Expose — discoverable capabilities

The foundation has shipped. WordPress 6.9 introduced the Abilities API — a PHP registry where core, plugins, and themes declare what they can do in a standard, machine-readable format. WordPress 7.0 added the JavaScript counterpart and wired abilities into the WP AI Client.

What’s still converging is the agent-facing layer: a foundational core/* ability set, Skills in the WordPress admin, and the problem of exposing thousands of abilities without drowning an agent’s tool selection. The shape holds either way. A capability declares its contract — name, description, schemas, permissions, implementation — so what WordPress can do is inspectable rather than asserted.

That’s the same contract Flavor Agent leans on when it exposes abilities to an MCP agent, instead of asking for trust in a black box.

Ring II · Site

Govern — owner-controlled use

Discovery alone is not trust. Once an agent can find a capability, the owner still has to decide whether it may use it — and see what happened after it did. AI 1.0.0, released 19 May 2026, shipped Request Logging, meant to surface AI activity across core, plugins, and themes, and Connector Approvals, which lets administrators decide which plugins may use a configured connector. A proposed Unified AI Management Layer for Core would consolidate permissions, usage metering, and capability-aware provider routing; the proposal recommends deny-by-default for cost safety while acknowledging the compatibility case for allow-by-default.

Flavor Agent turns that into owner-controlled review gates, not a policy claim.

A gap I hit in my own work

I ran into the edge of this myself. Request Logging can’t yet see a provider that bypasses the SDK’s HTTP transporter (#732) — which is exactly what the local sidecar in my ai-provider-for-codex project does, so those calls never reach the log. The point isn’t that the log is complete today. It’s that accountable AI use needs an inspectable record, and that a gap like this should become visible engineering work instead of disappearing behind a trust claim.

Ring III · Artifact

Attest — verifiable provenance

Governance stops at the site boundary. Once an output leaves WordPress, the evidence for where it came from and whether it’s intact has to travel with it or stay recoverable.

The difference from governance is structural. A governance log gives the owner an internal record. A signed manifest lets an outsider bind a claim to an artifact, identify who signed it, and detect any later change — whether that signer deserves trust is a separate question. It doesn’t make the content true. It makes the claim, and every change after it, testable.

Open C2PA proposals cover detecting existing Content Credentials on upload and signing new provenance claims onto published text and images. Durable credentials add bindings like fingerprints or watermarks, so that when the embedded metadata is stripped, you can still recover a manifest stored remotely. This lines up with Article 50 of the EU AI Act, scheduled to apply from 2 August 20261, which requires providers of certain generative systems to mark their outputs in machine-readable form so artificial or manipulated content stays detectable. C2PA is one way to implement that — not the legal requirement itself.

That direction is already visible in the provenance work landing in WordPress core.

The principle underneath

The external-instrument test

The same logic runs well past software: a claim gets more credible the moment it survives an instrument the claimant does not control. AI Leaders sharpened that for me through impact measurement. Blake Bertuccelli-Booth’s comparison of workforce programs against incubators made it concrete — administrative wage records and long-running randomized trials on one side, self-reported metrics bent by survivorship bias on the other.

Real impact is whatever survives an external instrument.

The external-instrument test

Read through that lens, the three rings are one accountability architecture. Each hands someone other than the author an instrument to read: a schema others can inspect, a log others can audit, a signature a stranger can verify. My ethics position sits on the same footing: trust isn’t a disclaimer appended after the model answers; it’s built into the workflow. I landed in the same place in cohort conversations about systems as far from WordPress as battery governance in rural Mongolia. Different domain, same requirement: inspectable decisions and accountable change.

I hold my own work to that standard. I don’t ask readers to trust my account of a contribution — I point to current review status, merged changes, and changelog lines, and I check GitHub rather than my notes. My portfolio’s proof bar renders state, never intent.

Some targets move faster than memory settles, so I match the source to the job: digests and search to discover, primary docs and changelogs for release facts, GitHub for live implementation state. Living roadmaps hold what no single source captures — the WordPress AI one runs a repository census on every refresh, because even an official board misses open pull requests. I ground the agents I rely on with Cloudflare AI Search so their answers stay anchored to retrieved sources, then I verify the load-bearing claims myself.

The same discipline shows up in what I ship. Suggestions stay review-first, applied changes leave an audit row, and a stale claim gets corrected rather than defended.

Bias belongs in that same accounting, not in a separate disclaimer. Model output carries the slant of its training and its prompt, so anything that touches a person or a truth claim stays review-first — a suggestion isn’t a judgment until a human has checked it against the source, the audience, and who gets harmed if it’s wrong. I don’t claim automated bias detection. Flavor Agent never certifies an output as neutral on my behalf. Where no reliable structural check exists, the honest move is to narrow the scope, not to perform a neutrality the system can’t guarantee.

Putting the friction back

Rapid change doesn’t push me to move faster or guess better. The tooling already does the relentless part — it will carry a half-formed idea all the way to something that looks shipped, without flinching. The scarce skill is knowing where to put the friction back: where a capability needs permission, an action needs a record, an artifact needs a verifiable claim, and my own conclusion needs checking. Every instrument here is exactly that — friction placed on purpose, wherever a claim should rest on evidence instead of trust. Mine included.
